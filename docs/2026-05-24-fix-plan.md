# Master Fix Plan — 6 轮体验官 49 Issue 的根因化简化方案

> **状态**：设计权威。所有 fix cluster sub agent 派单基于此文档。
>
> **配套**：体验官原报告见 `docs/2026-05-24-experience-rounds.md`。
> 体验官给出 49 Issue + 17 反馈 + 38 e2e 候选，本文档把它们**收敛到 10 个根因**，
> 按"克制熵增、用更少抽象、避免特殊逻辑"的原则给出统一简化设计。

## 顶层设计哲学

6 轮报告读完后，所有 Issue 背后归结为 **3 条更深的契约缺失**：

### 契约 1：**接口契约 explicit 化**（禁止"约定但不强制"）

OOC 多处用"约定俗成 + 文档说明"作为接口契约——实现者可能漏，新增者大概率漏。
**简化方向**：把约定升级为接口强制——类型签名 / 注册 hook / 启动期校验。

适用根因：#1（sediment 必须有 activates_on）/ #2（HTTP 必经 versioning）/ #3（frontend 不假设 backend 路径）/ #4（window 自带 renderXml）。

### 契约 2：**失败 explicit 由 source 报告**（禁止依赖下游 render 报错）

OOC 多处 bare `catch{}` / `void result` / "exec 视为成功 + render 内联 `<error>`"——失败被语义吃掉，LLM 看不到。
**简化方向**：所有 catch 块必须做 (a) 重抛 / (b) 写 event / (c) console.warn 至少一项；exec 层错误不依赖 render 暴露。

适用根因：#6（silent-swallow ban）/ #8（错误模型）。

### 契约 3：**状态翻转有唯一 owner**（删除兜底/补丁路径）

OOC 多处状态翻转有多个并存路径（HTTP + versioning / worker 周期扫 + 事件驱动 / catch + 兜底假设）。
**简化方向**：每个状态变化只有一个入口；删兜底逻辑。

适用根因：#2（HTTP 必经 versioning）/ #5（worker 只跑队列）/ #7（rollback 在 persistable 层）。

## 10 个根因（按严重性排序）

### 🔥 根因 #1 — dogfooding 协议级契约缺失（哲学承诺级，最高）

**涉及 Issue**：R4 #18 sediment 无 activates_on / R4 #19 子→父 reply 通道不可发现 / R4 #20 父侧 do_window 永不 archive / R4 #23 end({result}) 静默吞 / R4 #27 super reflection 幻觉过往。

**根因分析**：reflectable + collaborable.do 是 OOC 的两个核心 dogfooding 承诺；执行层缺协议契约：
- super flow 写 sediment 没有 schema enforcement（LLM 想写啥写啥）
- do_window 子→父回报通道在子侧 LLM 视野中只有 transcript 无 reply 协议（LLM 只能 hallucinate `end({result})`）
- `end` command 接受 `args.result` 但静默丢弃（不报错）

**简化设计**（契约 1：接口 explicit）：
1. **reflectable basicKnowledge** 显式列出"sediment write contract"：每个 `pools/<self>/knowledge/memory/<slug>.md` 必须含 frontmatter `title / description / activates_on`；同时把"activates_on 至少含一项"作为 reflectable 协议知识强制告诉 LLM
2. **do_window basicKnowledge 拆双视角**：基于 `is_creator_window` flag（thread 看到的 do_window 是父侧还是子侧），子侧 prompt 显式说"想把结果带给父，使用 `creator_do_window.continue({result})`"；删除"约定但不教"的黑箱
3. **end command 接受 result 参数**：扩 `end.args = { reason?, summary?, result? }`；`result` 时自动同步到 creator do_window transcript + auto-archive 父侧 do_window；删除"静默吞 result"的代码

**反熵措施**：
- 不引入"reply protocol" 新抽象——复用现有 continue command
- 不引入"sediment validator" 新模块——把检查写进 reflectable basicKnowledge 让 LLM 内化

### 🔥 根因 #2 — HTTP 写入 vs git-versioning 语义未定义（架构级）

**涉及 Issue**：R3 #12 HTTP 创建 stone / 改 server-source 不入 git / R5 #29 HTTP create stone 后 openMetaprogWorktree 看不到（dogfooding 第一步崩）。

**根因分析**：当前两个 stone 写入入口（HTTP service.ts 直写 vs stone-versioning.ts），无共享底层语义。HTTP 完全绕过 git-versioning，导致 worktree 看不到 stone、git history 缺。

**简化设计**（契约 3：状态翻转唯一 owner）：
- **所有 stone 写入必经 stone-versioning**——HTTP service.ts 不再直接 writeFile/mkdir，而是调用 `stone-versioning.openMetaprogWorktree → write → commit → tryMergeSelf` 一气呵成
- 对 bootstrap 性质操作（createStone）：HTTP 路径开一个 token=`bootstrap`的隐式 metaprog branch，立即 commit 到 main（ff merge）
- 对 evolution 性质操作（PUT server-source / self.md / readme.md）：HTTP 路径开 metaprog branch，commit，根据 path 划界自动 ff merge 或开 PR-Issue

**反熵措施**：
- 不引入"auto-commit" 旁路——HTTP 是 stone-versioning 的薄包装
- 不引入"uncommitted working tree" 半成品状态
- 删除 service.ts 中的直写代码路径（用更少代码替换更多代码）

**写入 meta**：`programmable.method_evolution.patches.http_must_go_through_versioning`。

### 🔥 根因 #3 — 前后端存储路径契约缺失（架构级）

**涉及 Issue**：R2 #6 tree marker 错位 / R3 #11 stones knowledge HTTP 写到 pools / R6 #39 ObjectClientRenderer 未加 main/objects/ / R6 #43 seeded stone 不在 sidebar tree（**4 处同根因**）。

**根因分析**：frontend 硬编码 backend 存储路径假设；2026-05-21 stones 重组（加 `<branch>/objects/`）后 frontend 各处 path-prefix 启发式失效。

**简化设计**（契约 1：接口 explicit）：
- **frontend 不假设路径，一律以 backend 字段为准**——backend 在 `/api/stones`、`/api/pools` 已经返回 `dir` 字段；frontend 只用 `item.dir`，不自己拼
- **删除 ObjectClientRenderer.clientAbsPath 自拼**——backend 加新 endpoint `/api/objects/{scope}/{id}/client-source-url` 直接返回绝对路径
- **删除 frontend path-prefix marker 启发式**——backend 在 `/api/tree` 响应里基于 `.stone.json` / `.pool.json` / `.flow.json` 元数据存在性计算 `marker`，frontend 只读 marker

**反熵措施**：
- 不引入"路径标准化"新抽象——backend 已经有 path，frontend 用就是了
- 不引入"path normalizer middleware"——删除前端 String 拼接（更少代码）

**写入 meta**：`app.client.doc.ts` 加 "frontend 路径解析必须经 backend resolver" / `app.server.doc.ts:modules.ui` 加 "tree marker 基于后端元数据权威"。

### 🔥 根因 #4 — render 缺 type-dispatch（visibility-first 横切）

**涉及 Issue**：R2 #1 skill_index XML 丢 skills / R2 #5 window 无 commands 元数据 / R2 #10 LLM 用错 command='talk' / R3 #15 talk_window 不携带 transcript / R4 #25 done thread 观测盲区 / R6 #46 render switch 缺 skill_index case。

**根因分析**：`src/thinkable/context/render.ts` 用"通用属性 only + switch case 偏分散"渲染——每个 window 只渲染共有字段；每种 window 自己的语义 payload 全部 silent drop；新增 type 容易漏 case。

**简化设计**（契约 1：接口 explicit）：
- **每个 window type 自带 `renderXml(window): string` 方法**——写在 `src/executable/windows/<type>/index.ts` 旁边
- **render.ts 退化为 100 行调度器**——按 window.type 查 registry，调对应 renderXml；无 switch case
- **registry 注册 hook**：window type 注册时同时注册 renderXml（不注册则启动期 fail-loud）
- 自动 surface：window.commands 由通用 base renderer 输出（每个 window 实现都不用自己写"commands 列表"）

**反熵措施**：
- 不引入"渲染策略"新抽象——`renderXml` 是 window 接口的一部分
- 不引入"context-aware renderer"——只是"window 自己知道怎么变 XML"
- 删除 render.ts 中已有 switch case（用接口替代）

**写入 meta**：`thinkable.context.render` 加"window type-dispatch 接口契约"段。

### 🔥 根因 #5 — worker 调度哲学错位（高）

**涉及 Issue**：R5 #30 worker 每 100ms 对 waiting thread 入队 → 10 jobs/s 线性膨胀；jobManager in-memory 无 cap。

**根因分析**：worker 周期扫所有 session 入队 job，把"调度发起"和"做 work 判断"耦合死。waiting thread 本应等事件唤醒，却被周期扫"兜底"入队。

**简化设计**（契约 3：状态翻转唯一 owner）：
- **worker 只跑队列**——不再扫 fs / 不再 enqueueOrphanRunningThreads
- **状态翻转（waiting → running）由事件源直接 enqueue**：
  - inbox 消息到达（talk-delivery / do-delivery）→ enqueue
  - issue comment / [resolved] / [recovery-needed] → enqueue
  - 显式 resume → enqueue
  - cross-object callee end → enqueue（通过 creator window 通知）
- **删除"兜底 enqueue"路径**——orphan running threads 是 design bug，应当根治在 readThread 持久化时刻而不是周期扫

**反熵措施**：
- 不引入"backoff" 旁路——根本不入队
- 不引入"dedupe" 复杂逻辑——根本不重复入队
- 删除 worker.ts 大量代码

**写入 meta**：`executable.server.worker` 加"事件驱动 enqueue / worker 只跑队列"哲学；`persistable.flow` 加"create flow object 不自动 kickoff，需显式触发"。

### 🔥 根因 #6 — silent-swallow 横切系统问题（高）

**涉及 Issue**：R4 #19 子→父无 reply / R4 #23 end({result}) 吞 / R5 #32 recovery-check catch{} / R5 #34 tryMergeSelf void remove 结果 / R5 #35 enqueueOrphanRunningThreads bare catch / R6 #44 open_knowledge 不校验 path（exec 视为成功，render 内联 `<error>`）。

**根因分析**：代码库散落 5+ 处 bare `catch{}` / `void result` / "exec 成功 + render 报错"。失败被语义吃掉，LLM 看不到。

**简化设计**（契约 2：失败 explicit）：
- **meta 加 silent-swallow ban 约束**：所有 catch 块必须 (a) 重新抛 / (b) 写 event / (c) console.warn 至少一项
- **exec 层错误不依赖 render**：所有 `open_*` 命令在 exec 时校验路径/存在性，错误直接走 command 错误链；render 不应出现 `<error>` 内联（除非 ContextWindow 自身就是 error state，那是合法的）
- **audit**：`grep -rn 'catch.*{}\|catch.*{[\s/]*}\|void [a-z]' src/` 扫所有同源 facet，逐个修

**反熵措施**：
- 不引入"error reporter"新模块
- 不引入"failure event bus"——OOC 已有 thread.events / Issue / debug 文件三个错误反映通道，每处 catch 选一个即可
- 自动化：可选加 ESLint rule 防回归

**写入 meta**：`observable` 节点加"silent-swallow ban"段。

### 根因 #7 — persistable 防御深度（高）

**涉及 Issue**：R5 #28 rollback() 不强制 supervisorAuthor / R5 #31 pruneStaleWorktrees 无 caller / R5 #34 tryMergeSelf 忽略 worktree remove。

**根因分析**：R12 "supervisor-only rollback" 校验只在 LLM 命令包装层；persistable 层 rollback() 接受任何字符串。pruneStaleWorktrees 注释承诺但 src 无 caller。

**简化设计**（契约 1：接口 explicit + 契约 3：唯一 owner）：
- **persistable 自 enforce R12**：`rollback()` 函数签名第一行 `if (supervisorAuthor !== SUPERVISOR_OBJECT_ID) throw new Error(...)`；任何 caller 越权都被拒
- **pruneStaleWorktrees 接入 bootstrap**：`ensureStoneRepo` 末尾自动调用
- **tryMergeSelf 不忽略 worktree remove 结果**：失败抛错并 log 上下文

**反熵措施**：
- 不引入"权限管理"新抽象
- 不引入"orphan worktree cleanup job"——直接每次 bootstrap 跑一遍

**写入 meta**：`persistable.stone-versioning.patches.r12_enforcement_at_persistable_layer`。

### 根因 #8 — 错误模型不统一（中）

**涉及 Issue**：R2 #7 404 vs 500 / R2 #8 Elysia validation 嘈杂 / R3 #16 endpoint 间形态不一致 / R5 #38 /health 500 vs 404 / R6 #45 sidebar tab 与 route 不一致 / R6 #49 code+message 自相矛盾。

**根因分析**：onError handler 设计只覆盖 AppServerError；Elysia native validation / not-found / 其他来源各走各的。

**简化设计**（契约 2：失败 explicit）：
- **onError 全覆盖**：handler 截获所有错误来源，统一转 `{error:{code,message,details}}` 包络
- **Elysia ValidationError → 压成 `{error:{code:"VALIDATION",message:<friendly>,details:<flat>}}`**
- **NotFoundError → `{error:{code:"NOT_FOUND",message,details}}` + HTTP 404**
- **删除"裸 `{code,message}`"返回路径**

**反熵措施**：
- 不引入"error transformer pipeline"
- onError handler 一处统一处理（已有，扩 coverage）

**写入 meta**：`app.server.doc.ts:bootstrap.errors` 加"onError 统一包络全覆盖"。

### 根因 #9 — activator union 不完整（中）

**涉及 Issue**：R4 #24 union 永不含 root / R6 #42 union 不接受 window-type-as-state。

**根因分析**：activator union 只接受 command_exec.paths + program.recent + knowledge.path；root 和"持续 open 的 window"不贡献 path。

**简化设计**（契约 1：接口 explicit）：
- **union 扩为**：`command_exec.paths + program.recent + knowledge.path + thread.contextWindows.where(status="open").map(w → w.type) + ["root"]`
- **"窗口存在性视为 implicit command_path"**——LLM 直觉：我在跟 alice talk 的整段时间都应该看到 talk 知识

**反熵措施**：
- 不引入"activation context" 新抽象
- 不引入"window-state activator"——只是 union 多加一项

**写入 meta**：`thinkable.knowledge.activator` 加 "union 包含 open windows 与 root"。

## 设计哲学总结

> **少抽象，不是更多特殊处理**——每个根因都是删代码、收紧契约、删特殊路径，而不是加特殊 case 修补症状。
>
> **可见，而非更多日志**——失败必须 source 报告，不依赖 render 兜底；状态翻转必须唯一 owner，不依赖周期扫兜底。
>
> **OOC 哲学一致性**——visibility-first / Object 自治 / 三分语义不是口号，而是接口契约的强制。
