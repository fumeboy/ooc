# OOC-4 L4.0：command → method 术语归一 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:subagent-driven-development / executing-plans。Steps 用 checkbox。
>
> **执行纪律**：执行 sub-agent **不要自己 commit**——每步写代码 + 跑 tsc/bun test 到全绿，Supervisor 逐步整合提交（带 co-author footer）。

**Goal:** 把 `command` 术语归一为 `method`（context window command = object method），**行为不变**。三层：①内部符号（tsc 兜底）②结构字段 commands→methods ③agent-facing「command」措辞 + exec 工具 arg key `command`→`method`。**保留 WindowType 字面量 `command_exec` 不动**（其改名 = 单列的 L4.0b，见 §尾）。

**Architecture:** 纯重命名增量。步骤 1-4 由 `tsc` 充当「漏改枚举器」（改符号 → 编译报错 → 逐个补齐）。步骤 5 是 agent-facing 文本/schema，**无编译兜底**，靠 exhaustive grep + 一致性 guard 测试 + 可跑的 e2e 验证（半改陷阱高发区）。

**Tech Stack:** TypeScript / bun；`git mv` 改文件名；`bun tsc --noEmit` 枚举漏改；`bun test src/` 回归（基线 1053 pass）。

**权威 scoping**：本 plan 基于 scoping workflow `wf_89675064-443`（5 agents 穷举四面）+ 其 synthesis。全量 maps 见该 workflow 输出。spec §4/§13。

### D2 **不改**（保留原样 — 保护清单，违反 = 破坏行为/parser/wire）
1. **WindowType 字面量 `"command_exec"`**（+ `CommandExecWindow` 接口、base/command_exec/ 目录、canonical id、`form.type==="command_exec"` 守卫、持久化）→ 全部留给 **L4.0b**（§尾）。牵动持久化兼容 + L4.1 canonical id 锚点 + ~14 处行为守卫。
2. **具体命令/方法名**（talk / do / say / continue / exec / refine / submit / end / add_step …）——只改容纳它们的字段名与 arg key，**名本身不变**。
3. **结构化触发 token `command::<window_type>::<command>`**（`triggers.ts:57` 字面 `expr.startsWith("command::")` 解析）——这是 wire token 不是散文措辞。出现在 `basic-knowledge.ts:246-247`、`reflectable-knowledge.ts:73`、`meta/object.doc.ts` 多处、`cookbook.author-ooc-agent.doc.ts:327`，并被 `triggers.test.ts:48`、`reflectable-knowledge.test.ts:98` 断言。**Task 5/6 的「command→method 措辞」必须跳过 `command::`**；改它 = 行为改（触碰 triggers.ts parser + 所有 activates_on frontmatter + 持久化 stones），出 L4.0 范围。
4. **HTTP wire 字段 `WindowTypeCatalogEntry.commands`**（`api.list-window-types.ts:31,51` + web `window-types.ts:20,66` + `ContextSnapshotViewer.tsx:453,459`）——前后端独立编译的 wire 契约，**L4.0 冻结为 `commands`**（如 web callMethod 般保留），改名延 L4.0b。注意 `def.commands`→`def.methods`（Task 2，内部读取，tsc 兜底）与这个**响应字段名** `commands` 是两回事：内部读取改、响应字段名留。
5. **command_exec 渲染 `form.command` 的 XML tag**（`command_exec/index.ts:51` `<command>`、`:55` `<command_paths>`、`:57` `<command_knowledge_paths>`）——渲染的是 D2#1 保护的 `form.command` 字段，**保留**（与 render.ts 列方法的 `<commands>/<command name=>` 不同，见 Task 5 区分）。
6. **policies.json 的 `commands` 键**（`permission-q0b.test.ts:255` 等 `JSON.stringify({commands:{...}})`）——这是权限 map 的同名异义键，**不是 window 定义**，不改（grep 排除项）。
7. **web `callMethod` / `call_method` / `/call_method` 路由**——已是目标名，不动。归一后「window method（exec 走 LLM）」与「object UI method（callMethod 走 HTTP）」同名，doc 须区分语义（Task 6）。

### D3 半改陷阱防线（步骤 5 强制）
exec schema key（`exec.ts`）、knowledge 协议文本（`basic-knowledge.ts` 等）、render hint（`render.ts`）、错误消息（command.refine/submit.ts）四处的 `command` 措辞**无任何编译约束**。必须：
1. 一次性全改（grep 穷举，见步骤 5 清单）。
2. 加**一致性 guard 单测**：断言 exec 工具 inputSchema 的 method arg key === 解析点读取的 key（schema↔parse 自洽）。
3. 跑可用的 agent-facing e2e（reflectable / programmable 等；hasLlmEnv 不可用则 skip-tolerant，但 guard 测试 + grep 必须过）。
