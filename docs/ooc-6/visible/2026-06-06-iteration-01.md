## 维度定位

Visible 描述 Object 持有并演化自身 UI 页面的能力——人类通过浏览器"看见"并与 Object 交互的那一面（自我塑造四件套之一：reflectable/programmable/**visible**/readable）。锚 `packages/@ooc/meta/object.doc.ts:4034` 节点 `dimensions.visible`（root.content 在 4036-4051）。

子能力：
- **stone client**：`stones/<self>/visible/index.tsx` 跨 session 稳定单页入口（meta `visible.stone_client`，约 4061 行）。
- **flow client pages**：`flows/<sid>/objects/<obj>/visible/pages/<page>.tsx` session 内多页扩展（meta `visible.flow_client_pages`，约 4080 行）。
- **ui_methods callable**：UI 经 HTTP callMethod 调 `executable/index.ts` 的 ui_methods，与 LLM 的 program.callCommand 分流（meta `visible.ui_methods_callable`，约 4101 行）。
- **client evolution**：super flow 用 write_file 改 *.tsx，下次客户端加载即热更（meta `visible.client_evolution`，约 4125 行）。
- **loop_timeline**：thread loop 的 Time Machine + window diff 可视化（meta `visible.loop_timeline`，约 4143 行）。
- **ooc:// 寻址 / agent-native 等价**：Agent 产出 `ooc://client/...` 稳定寻址 URI，1:1 映射 SPA route（meta root.patches.agent_native_parity，约 4259 行）。

## 本次迭代记录

主题：**client-source-url endpoint 接入 worktree 预览（commit `881c800c`，2026-06-06）**。

**动机**：stone/flow 升级为 session-worktree 统一模型后，业务 session 在自己的 worktree 里改 `visible/index.tsx`。若预览 endpoint 恒读 main canonical，session 改完的 UI 产物在控制面看不到——"自己改的界面自己看不见"破坏 visible 的元编程闭环。

**设计**：stone scope 带 `?sessionId` 时经 `resolveStoneIdentityRef(read)` 路由——已建 worktree 读 worktree 的 `visible/index.tsx`，未建/super flow/控制面（不带 sessionId）读 main canonical。read 模式保证不会为纯预览意外创建 worktree。

**实现**：`api.client-source-url.ts:57-63` —— `const stoneRef = sessionId ? await resolveStoneIdentityRef({baseDir, sessionId, objectId}, "read") : { baseDir, objectId }; absPath = join(visibleDir(stoneRef), "index.tsx")`，再 fallback legacy `client/index.tsx`（`:64-71`）。这是「接续指南 §2 五通道」中 visible endpoint 通道（通道五），与 program shell `$OOC_SELF_DIR`（通道一）等同源收敛到 worktree。

**验证**：`api.client-source-url.test.ts:65-94` —— main 上 seed 并 git commit canonical visible（worktree 从 main HEAD checkout），session s1 建 worktree 改写为不同内容；断言带 `?sessionId=s1` 命中 worktree 的 absPath，不带 sessionId 命中 main 的 absPath。配套 `:96-106` 钉死「stone 根具名 .tsx（如 Card.tsx）→ 404」契约。

> 配套前置修复 commit `378ecaf0`（2026-06-05）：根因不在 endpoint，而在 agent-facing basic-knowledge 引导仍指 legacy `client/index.tsx` 且含糊致 agent 误写根具名文件→首产页 404；改 `thinkable/knowledge/basic-knowledge.ts` 指向 canonical `visible/index.tsx` + 显式警示文件名固定/default export。两 commit 合起来完成 visible 首产页可渲染的完整闭环。

## 边界与未决

边界（不做什么）：
- 仓库**不提供生产级客户端渲染 host**：OOC 只写 tsx 文件 + 给读写接口与 fsUrl，真正打包/渲染由消费方（当前 web 控制面 vite dynamic import）实现（meta warnings，4677 行）。
- endpoint 坚持 **canonical 单入口契约**：只解析 `visible/index.tsx`（+legacy `client/index.tsx`），不扫 stone 根具名 tsx，杜绝入口歧义。
- visible 只管 UI 资源（tsx）+ 调用通道；方法库本身（含 ui_methods 实现）归 programmable。

未决（meta `dimensions.visible.todo`，4672 行）：
- **agent-native parity 缺口**：ui_methods 仅经 HTTP 暴露给前端，agent 端无等价 tool 路径——parity 公理下的显式技术债（root.patches.agent_native_parity，4259 行；loop_timeline 的 server method 等价也列后续 phase）。
- loop_timeline 的 `type_dispatch_diff_renderer`（per-window-type diff 渲染器）、flow_client_pages 在控制面的导航入口等增量项仍属演进面。
