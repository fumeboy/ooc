# 协作（collaborable）能力维度 · 迭代 01（2026-06-06）

## 维度定位

collaborable 定义 Object 之间如何协作——不是"调用对方的函数"，而是"消息 + 持续会话窗口"，所有跨 thread / 跨 object 的影响都经显式 inbox/outbox 与窗口，thread 间不共享内存（`packages/@ooc/meta/object.doc.ts:1740` "OOC Agent collaborable 概念" 节点）。在对象关系三轴中主要承载 **peer 平等轴**：同级 Agent 平等协作，只能 talk 说服、不能支配对方。

子能力：ThreadMessage 消息模型 / do_window（同 object fork 子线程）/ talk_window（跨 object 持续会话）/ talk-delivery（跨 object 派送统一入口）/ creator window（恒在通道）/ inbox/outbox（跨 thread 影响的唯一通道）。PR-Issue 是设计期 stone-versioning 的评审 token，不在 collaborable 运行时通道内（`object.doc.ts:2138`）。

## 本次迭代记录

**inbox 拆独立 per-message 存储，根治并发回报丢正文竞态**（commit `917db9f5`）

- **动机**：caller 同时被多个 callee 回报时，第二路 talk 正文静默丢失，inbox 只剩 framework 的 done 占位——peer 平等轴的多对一回报通道出现数据丢失。
- **根因/设计**：inbox 此前随 thread.json 整体 read-modify-write。worker 持 caller in-memory 跑很久（含 LLM），期间外部 `deliverTalkMessage` append 的新消息，被 worker 跑完的整体 `writeThread` 覆盖；per-thread 锁锁不住长 runJob。利用 inbox 本就是 append-only（消费靠 `consumedMessageIds` 派生过滤、无物理移除）的性质，把 inbox 拆成 `<threadDir>/inbox/<msgId>.json` per-message 文件：不同 msgId 不同文件→并发 append 互不覆盖；写盘只增不删→stale in-memory inbox 不会抹掉并发新增。
- **实现**：新增 `packages/@ooc/core/persistable/inbox-store.ts:34`（`persistInboxMessages` 幂等 append、`readInboxMessages:58` 扫目录合并损坏文件跳过）；`thread-json.ts:73` writeThread 先持久化目录、`stripVolatileForPersist`（`thread-json.ts:45`）从 thread.json strip inbox；`thread-json.ts:136` readThread 以目录为权威、merge 历史 thread.json.inbox 平滑迁移。delivery/worker/scheduler/do/service 等写入点零改动——仍 `thread.inbox=[...]` + writeThread，持久化层透明改走目录。
- **验证**：`packages/@ooc/core/persistable/inbox-store.test.ts:38` 三个并发回归（stale write 不覆盖 / 5 路并发全存活 / 幂等）；tsc + silent-swallow + deprecated gate 干净；persistable 170 + executable 370 + runtime 199 + app-server 105 全绿。

## 边界与未决

- **不做**：不调用对方函数、不直接改 peer/child 运行时状态（运行时管控走 talk 说服，见 `object.doc.ts:4907`）；改 seed 的 self/cross 走 stone-versioning（PR-Issue），不在 collaborable 运行时通道里（`object.doc.ts:2141`）。
- **不做**：thread 间不共享内存，所有跨 thread 影响必经 inbox/outbox（`object.doc.ts:1746`）。
- **未决**：inbox 目录权威 + 历史 thread.json.inbox 的 merge 是平滑迁移过渡逻辑（`thread-json.ts:138`），存量 world 迁移完毕后可移除该 fallback 分支。
- **未决**：ThreadMessage canonical vs legacy 别名兼容（`content ?? text`、`toObjectId ?? targetObjectId`）仍为存量数据保留，新写入强制 canonical（`object.doc.ts:1802` field_alias_compat）。
