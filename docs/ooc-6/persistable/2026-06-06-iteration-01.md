# 持久化（persistable）能力维度 · 迭代 01（2026-06-06）

## 维度定位

persistable 定义 Object 跨 session 的持久化能力——离开内存进程后还能从磁盘恢复成同一个 Object，看见上一次的所有沉淀。它不是数据库层，而是 Object 的「骨架与肉身」。

子能力（World 级三分 + 横切支撑）：

- **stone**：长期身份 + 设计源码（自我五件套），持久 + git 版本化。
- **pool**：事实型数据（data csv / sediment knowledge / files），持久但不版本化。
- **flow**：单次会话的运行产物（thread.json / data.json / debug），ephemeral。
- **ref 抽象 + 序列化策略**：四类 ref 集中路径计算；写盘前剥 in-process 内存字段。
- **stone-versioning / session-worktree**：身份演化的 git 工作流。

## 本次迭代记录：stone identity 收敛到 session-worktree 统一模型

### 动机

旧模型用 plain overlay（shadow）承载业务 session 对自我五件套的试验性修改：读时把 overlay 叠加在 stone 之上。问题是裸读路径（program shell `$OOC_SELF_DIR`、visible endpoint）看不到完整 identity——overlay 是部分文件的 shadow，不是完整副本，多个访问通道各自手拼路径，结构上一漏接就读到旧版/裸 main。

### 设计

把「业务 session 的 identity 试验」从 overlay shadow 改为一个完整的 git worktree 分支 `session-<sid>`（从 main HEAD lazy 派生）。三条规则（patch `main_worktree_evolve_model`，object.doc.ts L2989）：

- **main = canonical**：Object 已提交的权威自我，唯一默认读源。
- **session worktree = 会话内试验层**：业务 session（非 super、非控制面）的 write_file / edit 落 `stones/session-<sid>/`，本 session 即时生效、main 不变、别 session 读旧版；worktree 是完整副本，读写收敛到同一目录、裸读看得到完整 identity。
- **super-flow evolve_self = 合入闸门**：commit session 分支 → ff-merge 回 main（署名 objectId）→ GC。session 分支即演化单元。

所有 identity 访问通道统一过 `resolveStoneIdentityRef/Dir`，结构上杜绝再漏接。

## 边界与未决

边界（不做什么）：

- persistable 不是数据库层，不持有调度/业务逻辑；observable / executable / thinkable 通过 ref + 函数调用访问磁盘，不直接拼路径。
- worktree 模型不覆盖 pool sediment（`pools/<id>/knowledge/**` 独立直写不进 git）；super flow 反思写 memory 直接落 main。
- executable 命令集 / 注册的 readable 全局 main-canonical，loader 通道不 per-session 路由（per-session 改命令集本就走 evolve_self→main→重注册）。

未决 / 跟进项：

- `isValidObjectId`（`programmable` versioning）regex 不允许 `/`，嵌套 child（objectId 含 `/`）暂无法用完整 objectId 自 metaprog；放开后须确认 self-scope 前缀对 nested 用 `nestedObjectPath`。
- user-review-supervisor-seed 与 parent 跨 object 改 child seed 的显式授权校验当前未实现（design-ahead-of-code），现仅靠 self-scope 路径前缀隐式表达层级修改权。
- data pool 仅有 csv 读写原语；queryRows / bulk update / 索引按需增量加。
