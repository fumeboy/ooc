# 观测（observable）能力维度 · 迭代 01（2026-06-06）

## 维度定位

observable 描述 Object 的可观测能力：每一轮思考的 LLM 输入输出、tool 调用、context 状态可记录、可查看、可暂停、可回放——只在 thinkloop 周围加观测点，不改变 Object 行为。两个消费方：人类面（控制面 debug / timeline / pause）与 agent 面（super flow 自观测，从属 reflectable.super 通道）。

子能力：LlmObservation（最近一次输入输出快照）/ loop-level debug 落盘 / PauseChecker（tool call 前介入点）/ ContextSnapshot（与 XML 同源结构化快照）/ windowsSnapshot（每轮 window content hash）/ log-aggregator + activity 快照（系统级运行时诊断）。

锚 `packages/@ooc/meta/object.doc.ts` 节点 `observable`（title「OOC Agent observable 概念」，约 L2154-2326；子节点 llm_observation / debug_files / pause / context_snapshot）。

## 本次迭代记录

**主题：可观测增强三件套——日志去重限流 + activity 快照端点 + harness 超时快照**（commit `5a8dc1a5` / `8bfcae81` / `00e4bbc3`，Part 1/2/3）。

**动机**：三维度 harness 跑 programmable 体验官超时时，服务端日志 370/370 行全是同一条 `[readThread] references missing object ... skipping`——无界重复把真信号淹没；且超时只标 TIMEOUT 盲黑盒，只能事后 tail 才能诊断。典型 observability 反模式：同类警告无聚合/限流，长跑卡住无随时可读的系统快照。

**设计**：
1. 收口刷屏——观测层提供单一 emit 点，按稳定 key 去重计数 + 限流输出，既不刷屏又保留「还在发生 + 共多少次」信号，并维护滚动 tally 供快照消费。
2. 系统快照——HTTP 端点一次读出服务端此刻全貌（在跑/排队 job + ageMs + 主导日志模式），把「盲等到超时再 tail」变成「随时一读即诊断」。
3. 闭环到 harness——超时前 server 仍存活时抓快照写进报告。

**实现**：
- Part 1 `5a8dc1a5`：新建 `packages/@ooc/core/observable/log-aggregator.ts`（零 import 叶子模块，persistable→它无环）；`packages/@ooc/core/persistable/thread-json.ts:173-251` 6 处 readThread 刷屏警告路由经 `observeWarn`，370× 收口为 ~6 行 + 总数可见。
- Part 2 `8bfcae81`：`service.ts:178-187` 加 `getActivity` + RuntimeActivitySnapshot/Job 类型；`api.activity.ts` 端点挂载到 runtime module；route-audit.e2e 加真 HTTP 注册用例。
- Part 3 `00e4bbc3`：`packages/@ooc/tests/harness/orchestrate.ts` 体验官超时时 `captureActivitySnapshot(port)`（curl --noproxy best-effort，失败返回 null 不阻断收尾），写 `<dim>.timeout-snapshot.json` + dashboard 备注列展示 running 数 + 主导日志模式。

**验证**：
- 单测覆盖去重/限流/snapshot/level 路由（`packages/@ooc/core/observable/__tests__/log-aggregator.test.ts`）。
- `service.test.ts:83-100` getActivity 用例：running ageMs + queued 无 ageMs + 主导日志模式（readThread.missing-object ×5）。

## 边界与未决

边界（不做什么）：
- observable 不持有调度或业务逻辑，所有写盘都委托给 `persistable.debug-file`，只决定「何时记、记什么」。
- 不画 UI——loop_timeline / LoopDiffView 等渲染属 visible；observable 只产数据（windowsSnapshot 等）。
- contentHash 等 debug 派生字段不进 thread.json，业务字段保持最小。
- LlmObservation 是单例（模块顶层），同进程内只反映最近一次调用；并发多 thread 谁后写谁覆盖，要按 thread 区分历史须用 loop-level debug 文件。

未决/跟进项：
- **agent 面 parity 缺口**：Agent 自读历史并据此调整仍是演化方向，目前主要靠 super flow 读落盘产物，缺独立成熟入口（object.doc observable.content / root.patches.agent_native_parity）。
- **csv health 诊断**（AgentOfExperience 2026-05-24 反馈）：csv-pool 不校验 row.keys 与 header 一致性，建议 observable 加启动期/reflectable 主动扫一遍的 csv 健康诊断（object.doc.ts:3460 known-limitation）。
- log-aggregator 限流阈值（首 3 / 每 100）为经验常量，未做按 level 或按场景可配。
