# 思考（thinkable）能力维度 · 迭代 01（2026-06-06）

## 维度定位

thinkable 描述 Object 的思考能力：与 LLM 交互、构造 context、运行 thread 与 thinkloop。锚 `packages/@ooc/meta/object.doc.ts:125`（节点 `thinkable` "OOC Agent thinkable 概念"）。

子能力（同节点 `children`）：
- **identity**：Object 的双面身份（`self.md` 对内 / `readable.md` 对外）。
- **llm**：把 OOC 内部 Responses-first item 模型适配到 OpenAI / Claude provider。
- **context**：Object 本轮可见的全部世界，由若干 ContextWindow（= Object in Context）组成。
- **knowledge**：Object 持有的 markdown 知识，按 `activates_on` trigger 渐进激活。
- **thread**：思考过程拆成可并行、可等待、可恢复的 Thread Tree。
- **thinkloop**：单 thread 内一轮「构造 context → 调 LLM → 执行 tool → 写事件」循环。

## 本次迭代记录：window::root trigger always-on（修召回断裂）

代表迭代 commit `72a5aa6c`（2026-06-05）。

**动机**：reflectable harness 评为 Bad——agent 在 super flow 沉淀的 memory 永不被召回，自演化 dogfooding 的核心价值落空。

**设计**：根因是契约与实现分歧。`reflectable-knowledge.ts:72` 把 `window::root` 文档化为「root window 每个 thread 都有，等价任何时候」，agent 据此用 `window::root` 作 `activates_on`。但 root 是 manager 提供的**虚拟隐式父**，从不 push 进 `thread.contextWindows`；`evaluateTrigger` 的 object case 扫 contextWindows 找 `type==="root"` 的 open window → 永不命中。决策：与其改沉淀契约，不如让 `root` trigger always-on 坐实文档承诺。

**实现**：`evaluateTrigger` object case 特判 `trigger.objectType==="root"` → 直接 `return true`，见 `packages/@ooc/core/thinkable/knowledge/triggers.ts:211`（注释链路说明在 :206-:210）。一处特判，不动 parse 路径。

**验证**：`triggers.test.ts` 新增用例——`window::root` 对空 `contextWindows` 也命中（`packages/@ooc/core/thinkable/knowledge/__tests__/triggers.test.ts`）；core 单测 821→823 全绿，tsc 干净。

## 边界与未决

**边界**：
- llm 模块只负责「如何请求模型」，不负责「模型能做什么」——后者由 executable 暴露的 tool / method 决定（`object.doc.ts:215`）。
- reasoning 不作为普通上下文反复喂回（`object.doc.ts:226` patch `reasoning_not_replayed`），只用于 debug/回放。
- context 自动衰减 / emergency guard 已退役（`budget.ts:14`）；预算只保留软硬阈值，level 推进交回显式 compress/expand。

**未决/跟进**：
- 无 `activates_on` frontmatter 的 pool knowledge 永不自动激活（「死知识」）；`72a5aa6c` 同期已加写入体检 warning（commit `0aa20bb3`），但仍依赖人工读 warning，缺统一巡检。
- derived 窗口此前不写回 `thread.contextWindows`，靠 transient `_renderedWindows` 兜底观测（`context/index.ts:287`）；mock 路径与真实渲染存在两套读取分支，长期应收敛。
