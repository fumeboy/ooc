## 本次迭代记录：metaprog 命令瘦身 + 改身体统一走 session worktree + evolve_self 转正

### 动机

旧模型中「改自己身体」和「建别人 / 改别人」分属两条并行路径：
- 改 self-scope identity → session worktree 写 → super flow evolve_self（已建）。
- cross-object（建新对象 / 改别人）→ `create_object` / `open_worktree` → metaprog worktree → 独立 commit → merge。

两路并行带来认知负担（LLM 需判断走哪条）和结构复杂性（两种 worktree 机制）。evolve_self 的 `must-pr-issue` 分支是一等路径，只是 cross-object 改动从未进入它（被 metaprog 截走）。

### 设计

**单一写路径**：LLM 在 session 内改任何 stone（改自己 / 改别人 / 建新对象）一律 `write_file`，落本 session worktree（`flows/<sid>/objects/<target>/`），不 commit main；**唯一合入闸门** = super flow `evolve_self` 调 `tryMergeSelf`：
- self-scope（只改 `objects/<self>/`）→ ff-merge 到 main。
- cross-scope（动了别人 / 建了新对象）→ `must-pr-issue` → `requestPrIssueReview` → supervisor `metaprog resolve` 合入。

**super flow 不直接写 stone**：super flow（`evolve_self` 命令）是合入闸门 + sediment pool 沉淀路径（写 pool sediment knowledge）；改身体（stone identity）的动作在业务 session 里做，super flow 只 commit+merge。

**metaprog 命令纯治理**：`open_worktree` / `commit` / `merge` / `create_object` 已删；保留 `resolve`（PR-Issue 评审决议）/ `rollback`（回滚 Object stone）。命令面 KNOWLEDGE 更新为「直接写文件即可」教学。

## 边界与未决

边界（不做什么）：

- super flow 自身（`evolve_self` + `end_reflection_reminder`）不在 session worktree 里直接改 stone——「改身体」的动作在创建 super flow 的那个业务 session 里由 LLM write_file 做；super flow 负责把那个 session 的 worktree 合入。
- `metaprog` 命令族的命名（是否改为 `govern`/`review`）是待拍板的 design 问题，未在本次改动。
- reflectable 不管「是否允许 cross-scope」的策略——策略在 supervisor 评审 PR-Issue 时决定；协议层不 deny。

未决 / 跟进项：

- **metaprog 命名**：命令族瘦身后只剩治理，`metaprog` 名实不符（设计文档 §6.4）；更名待 Supervisor 定。
- **sediment 写入期校验**（继承自迭代 01）：frontmatter 靠 LLM 自觉，写错仅 warn 跳过；无写入期闸门。
- **cross-scope 合入粒度**：`tryMergeSelf` 整体分类（一文件越界 → 整 session 走 PR-Issue），不做文件级拆分（YAGNI，设计文档 §6.1）。
- **supervisor 自审 PR-Issue**：supervisor 改别人 / 建对象时自开自审是合法路径（versioning.ts:26），但无额外授权校验——仅靠 caller 身份是 supervisor 的校验（`method.metaprog.ts:122`）。
