# 反思（reflectable）能力维度 · 迭代 01（2026-06-06）

## 维度定位

Reflectable 描述 Object 的**自我反思 + 元编程触发协议**：通过一条名为 `super` 的特殊 session，把 Object 引到专用反思线程上，改写自身 stone 的身份文件（self.md / readable.md）与 pool 中的 sediment knowledge（memory / relations），下一轮新 thread 自动看见落盘内容、行为随之自我演化。

子能力（锚 `packages/@ooc/meta/object.doc.ts` `reflectable` 节点，约 L2400-2658）：
- `super_session`：受保护 sessionId="super" 反思通道（L2453）
- `super_alias_target`：talk_window.target="super" 翻译为指向自身 super 分身（L2472）
- `reflectable_knowledge`：进入 super flow 自动注入协议知识（L2493）
- `end_reflection_reminder`：业务 thread 调 end 时的反思提醒（L2523）
- `memory_layout`：长期记忆落盘位置 + sediment write contract（L2561）
- `metaprogramming`：业务线程→super→write_file→下轮生效的元编程闭环（L2627）

边界（root.content L2412-2415）：reflectable 只动 sediment（运行时沉淀），不直接改 seed knowledge（先天能力基底走 PR-Issue）；改身体（executable / visible）的形状由 programmable / visible 承担。

## 本次迭代记录

**主题：evolve_self 身份合入重做为「session worktree 即演化单元」（commit db9e54ea，2026-06-06）。**

### 动机
旧 plain-overlay 模型下，业务 session 对自己 identity 的改动落 shadow overlay，evolve_self 时需逐文件读 overlay 再应用进一个新建的实验 worktree——双目录、shadow 不可裸读、合入路径绕。worktree 统一模型（doc §4）要求业务 session 的读写都收敛到单一 `stones/session-<sid>/` 完整副本，evolve_self 随之不能再读 overlay。

### 设计
session 分支即演化单元：不再「读 overlay 逐文件应用」，而是直接 commit 业务 session 已有的 `session-<sid>` worktree 分支 → rebase 到 main → self-scope ff-merge 回 main → GC（移除 worktree + 删分支）。署名 = objectId（非 bootstrap），契合 self-scope 自治区 ff-merge；冲突 / 越界由 `tryMergeSelf` 上抛，worktree 保留、main 不变（fail-loud）。

### 实现
- `packages/@ooc/core/programmable/evolve-self.ts:98-117`（`evolveSelfDiff`）：用 `gitStatus` porcelain 列工作树改动，`porcelainLineToRel`（L83）映射回相对 object 自治区的 relWithinObject；worktree 未建→空数组。
- `packages/@ooc/core/programmable/evolve-self.ts:124-207`（`evolveSelfMerge`）：复用 `commitWorktree`（versioning.ts:263）+ `tryMergeSelf`（versioning.ts:346）+ `requestPrIssueReview`（versioning.ts:406）；merged 后删 session 分支收尾（L170），防御性 must-pr-issue 转 PR（L185）。
- 删旧 files 子集选择（whole-session 合入）、`NO_OVERLAY`→`NO_CHANGES`；命令面 `packages/@ooc/builtins/root/executable/method.evolve-self.ts:110,129` 改调新 core。

### 验证
- `packages/@ooc/core/executable/__tests__/evolve-self.test.ts:108-124`：diff 列 `["self.md"]` → merge `kind==="merged"`、main self.md 变为 evolved 内容、`gitLastAuthor`==="alice"（署名 objectId 非 bootstrap）。
- 多文件场景 L150,158：diff/merge files 均含 `executable/index.ts` + `self.md`。
- commit 自述：file-overlay-redirect / write-file.versioning / context P2 / evolve-self / program OOC_SELF_DIR 全改写为 worktree 模型，core+builtins 全绿，tsc/silent-swallow/deprecated 全绿。

## 边界与未决

边界（不做什么）：
- super 是 self-scoped：只观察 / 修改 Object 自己；cross-object 改别人子树走 PR-Issue，super 本身从不跨 object（object.doc L2437-2440）。
- reflectable 只管「为什么 / 何时 / 在哪条线程」做元编程；被改对象的形状归 programmable（方法）/ visible（UI）。
- super flow 不开新业务任务（不跑 program shell 改外部文件、不 file_window.edit 业务代码）；不直接改 seed knowledge / executable / visible / .stone.json（高赌注走 stone-versioning）。

未决 / 跟进项：
- sediment write contract 依赖 LLM 自觉写合法 frontmatter；写错 schema 仅 console.warn 跳过该篇（fail-loud 但不阻断），无写入期校验闸门。
- `evolveSelfMerge` 的 must-pr-issue 分支被标注「理论上不该越界」的防御性路径——worktree 只装自身 identity 时不会触发，cross-scope 走 metaprog worktree，二者职责边界尚未在 doc 完全统一表达。
- end_reflection_reminder 阈值门控（thread.events.length > N）默认未启用，简单 thread 也会提示，靠 LLM 自判是否反思。
