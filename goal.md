# OOC 系统重构任务说明

## 背景

当前正在对 OOC 系统进行重构。

- 原系统目录：`~/x/ooc/ooc-0`
- 原系统文档入口：`~/x/ooc/ooc-0/docs/meta.md`
- 新系统文档入口：`~/x/ooc/ooc-2/meta/index.doc.js`

本次重构不是对旧系统的机械搬运，而是一次面向长期演进的工程化收敛：保留真正有价值的核心能力，去掉导致系统持续失控的实现方式和复杂度来源。

## 原系统主要问题

- 代码长期由 Agent 自由生成，缺少稳定的架构约束与质量监督。
- 文档、设计、实现之间缺少强绑定，容易出现文档与代码脱节。
- 模块边界不清晰，功能不断叠加，系统复杂度持续膨胀。
- 当复杂度超过可维护阈值后，新增需求与重构成本都会快速上升。

## 重构目标

1. 采用“文档即系统组成”的方式组织新系统文档。
   - 使用 JavaScript 编写文档。
   - 通过 `import` 引用相关文档与源代码。
   - 让文档结构、代码结构、系统结构尽可能保持同步。

2. 避免“孤儿文档”与“孤儿代码”。
   - 文档中描述的能力，必须能映射到实际代码。
   - 代码中保留的核心能力，必须在文档中有明确位置和说明。

3. 从原系统中提炼最小核心能力集合。
   - 优先抽取系统成立所必需的核心能力。
   - 不直接复制旧系统的全部设计与实现。
   - 以“更少概念、更清晰边界、更低耦合”为目标重新组织系统。

4. 以可维护性为第一约束控制复杂度。
   - 新设计必须优先保证可理解、可修改、可验证。
   - 若一个能力无法被清晰解释、清晰归属、清晰验证，则不应直接迁入新系统。

## 重构原则

1. 文档先行，代码随后。
   - 任何模块迁移前，先确认其在新文档体系中的位置、职责、边界与依赖。

2. 小步迁移，持续收敛。
   - 每次只迁移一个足够小、足够清晰的模块或能力单元。
   - 每完成一次迁移，都应让系统更简单，而不是更庞杂。

3. 先核心、后外围；先简单、后复杂。
   - 优先迁移“核心且复杂度较低”的模块。
   - 暂不优先处理高耦合、边界模糊、依赖链过长的模块。

4. 设计必须显式，禁止隐式补全。
   - 如果新系统文档中没有说明某项机制、边界或约束，不应自行脑补实现。
   - 文档未覆盖的内容，必须先与用户确认，再决定是否迁移或如何迁移。

5. 每次迁移都要可追踪、可解释、可验证。
   - 能说明为什么迁。
   - 能说明迁移后的职责边界。
   - 能说明相对旧系统删掉了什么、保留了什么、简化了什么。

## 可参考的重构经验

以下经验适用于“旧系统已经可运行，但复杂度过高，需要提炼核心并重建边界”的重构任务。

1. 不要搬运旧系统，要重建系统边界。
   - 重构的重点不是把旧代码换个目录重新写一遍，而是重新回答“系统最小需要什么”。
   - 任何不能证明必要性的机制，都不应默认迁移。

2. 先做减法，再做加法。
   - 大多数失败的重构，不是因为能力不够，而是因为舍不得删。
   - 与其把旧系统全部保留后再慢慢清理，不如先只保留最小核心闭环。

3. 优先迁移高价值、低耦合、易验证的能力。
   - 最适合作为首批迁移对象的，通常是核心能力中边界最清晰的一部分。
   - 不要一开始就处理依赖链很长、历史包袱很重的复杂模块。

4. 先证明骨架成立，再补功能。
   - 先建立最小可运行结构，再逐步补充能力。
   - 新系统必须先证明“基本架构是成立的”，再追求覆盖更多旧功能。

5. 文档不是说明书，而是架构约束的一部分。
   - 文档应该先定义模块为什么存在、边界在哪里、与谁协作。
   - 如果这些问题文档回答不清，代码实现通常也会继续失控。

6. 每次迁移都必须明确取舍。
   - 需要明确说明本次迁移保留了什么、删除了什么、为什么这样取舍。
   - 如果只是“把旧功能重新实现”，而没有减少概念和机制，通常不算有效重构。

7. 不要让兼容层无限生长。
   - 兼容、过渡、临时桥接机制可以存在，但必须被视为短期手段。
   - 如果兼容层长期堆积，新系统会重新长成旧系统的样子。

8. 任何未定义内容都不应由实现者自行脑补。
   - 文档没定义、边界没说清、约束没写明的地方，默认先暂停。
   - 这类问题必须先与用户确认，再决定是否实现以及如何实现。

9. 每次迁移都要能被验证“系统是否更简单”。
   - 判断标准不是代码行数是否减少，而是是否更容易理解、更容易定位、更容易验证、更少特殊情况。
   - 如果迁移后只是换了写法，但边界没有更清晰，通常说明重构并未真正完成。

10. 把长期维护成本当成第一等公民。
   - 判断一个设计是否成立，不只看当前能否实现，还要看后续是否容易交接、调试、扩展与删除。
   - 如果未来的维护者难以快速理解该模块，该设计就还不够成熟。

## 重构自检问题

每次准备迁移一个模块或能力前，至少先回答以下问题：

- 它在新系统里为什么存在？
- 它的最小职责是什么？
- 它的边界是否能用几句话说清楚？
- 它依赖哪些模块，是否可以进一步减依赖？
- 如果暂时不迁它，系统会失去什么？
- 它迁入后，系统是更简单了，还是只是换了一种复杂写法？

## 推荐迁移流程

1. 阅读上下文。
   - 先阅读旧系统相关实现与旧文档。
   - 再阅读新系统文档入口，确认该能力在新体系中的归属。

2. 定义迁移对象。
   - 明确本次迁移的是“一个模块”“一个能力”还是“一个最小闭环”。
   - 明确输入、输出、依赖、边界和非目标。

3. 先做文档映射。
   - 说明该能力在新系统中为什么存在。
   - 说明它与周边能力的关系。
   - 说明它对应哪些文档与代码文件。

4. 再做实现迁移。
   - 优先复用旧系统中已经被验证过的核心思想。
   - 不继承旧实现中的历史包袱、偶然复杂度和模糊抽象。

5. 最后做收敛检查。
   - 检查文档是否覆盖了实现。
   - 检查实现是否仍然符合新文档。
   - 检查是否引入了新的无主概念、重复机制或不必要层次。

## 单次迁移完成标准

一次迁移可视为完成，至少应满足以下条件：

- 新系统文档中已经明确说明该能力的职责、边界和位置。
- 相关文档与源代码之间已经建立明确引用关系。
- 没有产生“文档里有、代码里没有”或“代码里有、文档里没有”的内容。
- 相比旧系统，实现复杂度是下降的，或者至少没有继续膨胀。
- 本次迁移的取舍理由是清楚的：保留什么、删除什么、为何简化。
- 若迁移过程中出现文档未定义事项，已经先与用户确认，而不是擅自决定。

## 非目标

以下事项不属于本轮重构的优先目标：

- 不追求一次性完整迁移整个旧系统。
- 不为了“功能看起来更全”而保留旧系统中的复杂设计。
- 不在文档尚未定义清楚时抢先实现大而全方案。
- 不引入新的抽象层，只为包装旧复杂度而不真正降低复杂度。

## 执行要求

- 每次开始迁移前，默认先检查新系统文档。
- 如果文档没有覆盖实现所需的信息，必须暂停实现并与用户仔细确认。
- 所有迁移工作都应服务于两个核心结果：
  - 新系统的文档与代码保持同步。
  - 新系统的核心能力更稳定，整体复杂度更低。
