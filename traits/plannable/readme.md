# 规划能力

复杂任务需要先拆解再执行。不要试图一步到位。

## 核心原则

**先想清楚要做什么，再动手做。**

"太简单不需要规划"是最常见的反模式。即使看起来简单的任务，拆解后往往会发现隐藏的复杂度。

## 什么时候需要规划

- 任务涉及多个步骤
- 你不确定从哪里开始
- 任务涉及多个数据源或对象
- 你需要和其他对象协作
- 任务的完成标准不明确

## 规划方法

### 1. 澄清目标

在开始之前，确认你理解了任务的目标：
- 最终产出是什么？
- 谁需要这个结果？
- 什么算"完成"？

如果目标不清楚，先向用户提问（`talk("...", "user")`），不要假设。

### 2. 用认知栈拆解任务

用 `create_plan_node` 把任务拆解为可独立验证的小步骤，所有步骤挂在根节点下平级排列：

```javascript
const root = createPlan("任务名称", "任务的具体目标描述");

// 所有步骤都挂在 root 下，平级排列
create_plan_node(root, "收集信息", "从 3 个来源收集数据", ["web_search"]);
create_plan_node(root, "分析数据", "对比各来源观点");
create_plan_node(root, "撰写报告", "整理结论并回复用户");
// moveFocus 到第一个待办节点开始执行
```

每个步骤应该：
- 有明确的完成标准
- 可以独立验证
- 足够小（一两轮思考能完成）
- 声明需要的 traits（让系统自动加载相关知识）

### 3. 按步骤执行

- 一次只做一步
- 每步完成后用 `finish_plan_node(summary)` 标记，focus 自动推进到下一步
- 验证当前步骤的结果后再进入下一步
- 如果发现计划需要调整，用 `create_plan_node` 添加新步骤或 `removeStep` 删除不需要的步骤

### 4. YAGNI 原则

不做没被要求的事：
- 不添加"以防万一"的功能
- 不做"顺便优化"
- 不解决没被提到的问题
- 当前任务需要什么就做什么

## Red Flags

- "这个很简单，不需要计划" → 拆解后再判断
- "我先把所有东西都做了再说" → 一次只做一步
- "顺便把这个也改了" → 不在计划内的不做
- 做了 3 轮还没有明确进展 → 停下来重新规划

## 常见的合理化借口

| 借口 | 现实 |
|------|------|
| "这个很简单不需要规划" | 简单任务是未检查假设最多的地方 |
| "我边做边想" | 边做边想 = 边做边返工 |
| "先写代码再重构" | 先写错再改 ≠ 规划 |
| "规划太慢了，直接开始更快" | 不规划导致的返工更慢 |
