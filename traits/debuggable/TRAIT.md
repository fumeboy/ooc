# 系统化调试能力

遇到错误时，你必须系统化地调查根因，而不是猜测和尝试。

## 铁律

**没有根因调查，不做任何修复。**

症状修复会制造新 bug。猜测修复会浪费轮次。时间越紧越要系统化，因为猜的成本更高。

## 四阶段流程

必须按顺序执行，不能跳过。

### 阶段一：根因调查

1. 完整读取错误信息（包括堆栈、输出、上下文）
2. 稳定复现问题（记录复现步骤）
3. 回顾最近的变更（什么改了？什么时候开始出错的？）
4. 从症状反向追踪数据流，定位出错的第一个环节

### 阶段二：模式分析

1. 找到代码库中类似的、正常工作的代码
2. 完整阅读参考实现
3. 逐项列出正常代码和出错代码的差异
4. 记录依赖和隐含假设

### 阶段三：假设与测试

1. 形成一个具体的假设（"我认为问题是 X，因为 Y"）
2. 做最小变更来验证这一个假设
3. 观察结果
4. 如果假设错误，回到阶段一形成新假设

### 阶段四：实现修复

1. 实现单一的根因修复
2. 验证修复解决了问题
3. 验证没有破坏其他功能
4. 如果修复失败且已尝试 3 次，质疑架构而非继续打补丁

## Red Flags

如果你发现自己在想以下内容，立即回到阶段一：

- "快速修一下先，之后再调查"
- "试试看这样行不行"
- "应该是这个问题"
- "我改了几个地方，看看哪个生效"
- "加个 try/catch 就好了"

这些都是猜测，不是调试。

## 常见的合理化借口

| 借口 | 现实 |
|------|------|
| "这个 bug 很明显" | 明显的 bug 往往有不明显的根因 |
| "没时间做完整调查" | 猜测修复浪费的时间更多 |
| "先修了再说" | 症状修复会掩盖真正的问题 |
| "改了好几个地方，总有一个对的" | 你不知道哪个生效了，也不知道其他改动引入了什么 |
