# 验证能力

## 🚪 完成前门禁（必读）

**你即将声明完成时，必须先回答以下问题：**
1. 你运行了什么验证命令？（必须是本轮执行的，不是之前的）
2. 输出是什么？（引用具体输出，不是"测试通过了"）
3. 输出是否支持你的结论？

**如果任何一项答不上来，先运行验证再 return。**

---


你不能在没有验证证据的情况下声明任何事情已完成。

## 铁律

**没有新鲜的验证证据，不做任何完成声明。**

违反这条规则的字面意思就是违反它的精神。没有例外、没有捷径、没有"我很确定"。

## 验证流程

每次你准备声明完成时：

1. 确定能证明你声明的验证方式（运行代码、检查输出、读取结果）
2. 执行验证（新鲜的、完整的）
3. 读取完整结果
4. 确认结果支持你的声明
5. 然后才能声明

跳过任何一步 = 没有验证。

## Red Flags

如果你发现自己在想以下内容，立即停下来去验证：

- "应该没问题"
- "我确信这是对的"
- "刚才改过了，肯定好了"
- "这个很简单，不会出错"
- "上次跑过了，应该还是通过的"

这些都是信心，不是证据。

## 常见的合理化借口

| 借口 | 现实 |
|------|------|
| "我很有信心" | 信心 ≠ 证据 |
| "刚才验证过了" | "刚才"不是"现在"，代码可能已经改了 |
| "这个改动太小了" | 小改动也会引入 bug |
| "验证太慢了" | 不验证导致的返工更慢 |
| "逻辑上一定是对的" | 逻辑正确不代表实现正确 |

## 适用场景

在以下任何情况下，验证都是强制的：

- 声明任务完成
- 声明代码能工作
- 声明问题已修复
- 声明数据已保存
- 向用户报告结果
- 任何形式的"成功"表述
