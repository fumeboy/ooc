## 一、前因后果（为什么做、依据什么）

- **设计权威**：`.ooc-world-meta/.../objects/supervisor/children/class/knowledge/object-model.md`（对象模型单一权威，核心 9 条）
  + 同目录 `example.md`（逐文件接口模板）。本轮任务＝把 `packages/@ooc/builtins/` 存量 class/object 改成与 example.md
  **完全一致**的接口模式，再反推 core。
- **核心转变**（旧 → 新）：
  - `self`：从「ContextWindow（id/class/title + 业务数据全平铺一坨）」→ 拆成纯 `Data`（types.ts）+ runtime 信封 + 独立投影态 `win`。
  - 方法：`exec(ctx)` 捆绑 `ctx.self/ctx.args` → `exec(ctx, self, args)` 三参分离；`self` 收窄为纯 Data。
  - 注册：副作用 `builtinRegistry.registerWindowClass(...)` barrel → 声明式 `index.ts` 的 `export const Class`。
  - readable：`(ctx)=>XmlNode[]` → `{ readable:(ctx,self,win)=>({class,content}), window:[{class,object_methods,window_methods}] }`。
  - constructor：返回 `{ok,window}` 的特殊 method → `Class.construct.exec(ctx,args)=>Data`（runtime 据 Data 包信封）。
  - package.json：去 `type`/`instantiate_with_new_world`/`members`，只留 `kind`(class/object)+`class`(继承)。
- **承重决策**（example.md 自陈「示意骨架非逐字」，矛盾/留白处由 supervisor 裁决，详见
  `docs/2026-06-15-object-model-migration.md` 第一节）：
  1. `self` = Data 本体（非 `{data}` 包装）；信封字段经 `ctx.object` 取。
  2. `types.ts` = 纯 Data；投影态 `win` 每 class 在 readable 自定义、与 Data 分离。
  3. constructor 取 `ConstructorContext`（无 `object`）、`exec(ctx,args)=>Data`、失败 throw；委托类 tool-object
     经 `ctx.runtime.instantiate('_builtin/<child>', args)`（`RuntimeHandle`）造子对象。
  4. **构造槽位键名是 `construct` 不是 `constructor`** —— JS `Object.prototype.constructor` 会遮蔽该键
     （`({}).constructor === Object` 恒真 → 单例无法识别；TS 也拿 `Function` 比对报错）。example.md 原写
     `constructor` 是该陷阱下的笔误，**已同步修正权威 example.md** + 全部落地代码。
  5. supervisor/user 按 object-model 迁移映射改为 **`kind:"object"`**（实例，非 class；`instantiate_with_new_world` 废弃）。
